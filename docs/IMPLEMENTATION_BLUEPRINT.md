## 2. 当前工程定位

当前仓库 `nursing-admin-v2` 的技术形态是：

- Next.js App Router 管理后台
- Tailwind + 自定义设计系统组件
- Demo 阶段，以前端页面与交互验证为主
- 已具备老人、设备、报警、健康、护理、员工、分析等后台页面雏形

因此，当前工程在平台中的定位应明确为：

### 2.1 当前主定位

**Admin 管理后台前端壳层 + 运营工作台。**

### 2.2 当前不建议在本仓库直接承担的能力

以下能力不建议直接堆在本仓库中：

- 家属 APP 全量逻辑
- 员工 APP 全量逻辑
- IoT 设备协议接入
- AI 模型训练任务
- 多服务后端领域逻辑

原因是这些能力属于独立应用或中台服务，应与 Admin 前端解耦。

## 3. 与平台总架构的对应关系

当前工程可映射到整体架构中的如下位置：

```text
应用层
└── Admin 管理后台   ← 当前仓库承担

业务中台
├── Elder Service
├── Care Service
├── Health Service
├── Staff Service
├── Device Service
├── Alert Service
└── Billing Service   ← 当前仓库消费，不直接承载

数据中台
├── 业务数据层
├── 健康时序层
└── AI / IoT 数据层   ← 当前仓库只展示结果

基础设施层
├── API Gateway
├── Message Bus
├── IoT Platform
└── AI Platform       ← 当前仓库只通过 API 接入
```

结论：

- 当前工程是“平台可视化入口”
- 不是“平台全栈承载体”
- 未来所有业务能力都应通过 API / BFF 进入当前 Admin

## 4. 当前 Admin 工程建议承接范围

基于现阶段 Demo 和已有页面结构，建议当前工程优先承接以下范围。

### 4.1 运营驾驶舱

包括：

- 首页 Dashboard
- 入住率、床位率、护理完成率、报警数、设备在线率
- 重点风险老人和关键告警列表

### 4.2 业务运营后台

包括：

- 老人管理
- 房间床位管理
- 入住与退住流程
- 入住登记 AI 分级与计划生成流程
- 员工排班与任务中心
- 护理服务配置
- 健康数据查看
- 设备总览 / 状态 / 资产管理
- 报警中心 / 报警历史
- 报表中心 / 系统设置

### 4.3 配置与治理后台

包括：

- 护理服务项标准化配置
- 报警规则配置
- 角色权限配置
- 组织 / 院区 / 楼层配置
- 设备台账与维保配置

### 4.4 AI 管理入口

当前 Admin 适合作为 AI 的运营与配置入口，而不是模型训练入口。建议展示：

- AI 风险预警结果
- AI 护理建议记录
- AI 问答日志
- AI 模型版本状态
- AI 规则启停开关

### 4.5 首条 AI + Admin 业务闭环建议

当前工程最适合优先落地的一条 AI 真实业务闭环，不是开放问答，而是：

**入住登记 AI 护理分级与计划生成。**

建议流程：

1. 管理人员在 `/elderly/checkin` 完成入住登记表。
2. 系统将基础信息、病史、用药、ADL、认知等结构化为评估输入。
3. AI Service 返回护理级别建议、原因摘要和计划建议。
4. 护理主管确认或手动调整护理级别。
5. Care Service 按级别和模板生成 `elder_care_plan` 与 `care_task`。
6. Notification Service 生成定时提醒，到点推送给护理人员。
7. 护理执行结果回写，作为后续 AI 再训练或规则优化输入。

这条链路的价值在于：

- 直接作用于业务主流程
- 可在 Admin 端闭环验证
- 能天然串联 Elder / Care / AI / Notification 四个服务域
- 能形成数据库、任务中心、消息中心和执行反馈的统一闭环

## 5. 推荐后端服务拆分

若后续进入真实实施阶段，建议后端按领域拆分，而不是直接围绕页面 API 开发。

### 5.1 领域服务建议

| 服务 | 核心职责 |
|------|----------|
| User Service | 用户、身份、登录、角色、组织归属 |
| Elder Service | 老人档案、家属关系、入住退住、标签 |
| Care Service | 护理计划、任务生成、服务打卡、交接班 |
| Health Service | 生命体征、病史、用药、健康报告 |
| Staff Service | 员工档案、排班、任务归属、绩效 |
| Device Service | 设备台账、状态、维保、在线监控 |
| Alert Service | 报警规则、事件流转、处理闭环 |
| Notification Service | 站内信、推送、短信、消息模板 |
| Billing Service | 账单、收费、套餐、支付、欠费提醒 |
| AI Service | 风险预测、问答助手、护理建议、异常识别 |
| IoT Service | 设备接入标准化、消息解码、设备事件入总线 |

### 5.2 Admin 与后端的交互建议

前端建议通过以下两种方式接入：

- 统一 API Gateway
- 针对复杂页面增加 BFF 聚合层

不建议当前前端直接分别耦合大量底层微服务地址。

## 6. 推荐数据分层

### 6.1 业务主数据层

建议存储：

- 老人主档案
- 家属主档案
- 员工主档案
- 房间与床位
- 护理服务定义
- 费用套餐定义

- PostgreSQL

建议存储：

- 入住退住记录
- 护理任务与执行记录
- 账单与支付记录
- 消息通知记录
- 报警处理记录

建议数据库：

- PostgreSQL

建议存储：

- 心率

- 血糖
- 血氧
- 睡眠

- TimescaleDB

建议存储：

- 登录态

- Redis

## 7. 推荐三端实施路径

### 7.1 第一步：先做 Admin 完整运营闭环

当前仓库优先做成一套可演示、可验证、可后接 API 的运营后台。

目标：

- 确立信息架构
- 确立设计系统
- 确立领域模型展示方式
- 确立导航和页面骨架标准

### 7.2 第二步：再启动员工 APP

员工 APP 与 Admin 应共享以下对象：

- 护理任务
- 报警事件
- 老人基本信息
- 健康录入结构

员工 APP 建议单独建仓，以 Flutter 为主。

### 7.3 第三步：家属 APP 独立建设

家属 APP 与 Admin 共享以下对象：

- 老人状态摘要
- 护理记录摘要
- 健康报告摘要
- 消息通知记录
- 视频探视预约记录

家属 APP 也建议独立建仓，以避免权限模型和交互模型污染 Admin。

## 8. AI 接入实施建议

### 8.1 当前阶段建议

当前仓库处于 Demo 阶段，AI 最适合先做展示闭环，而不是先做模型闭环。

建议先落：

- AI 风险卡片
- AI 护理建议卡片
- AI 问答助手页面
- AI 事件解释面板
- AI 入住分级建议面板

### 8.2 下一阶段建议

当后端能力成熟后，再接入真实 AI 服务：

- AI 风险预测 API
- RAG 家属问答 API
- 护理建议推荐 API
- 异常检测 API
- 入住评估与护理级别推荐 API

### 8.3 Admin 需要承载的 AI 页面类型

- AI 风险总览页
- AI 推理详情页
- AI 模型配置页
- AI 规则启停页
- AI 问答日志页
- AI 入住评估确认面板

## 9. IoT 接入实施建议

### 9.1 当前阶段建议

当前 Admin 先展示 IoT 的业务结果，而不是直接实现设备协议。

建议页面关注：

- 设备总数
- 在线率
- 电量 / 信号
- 告警事件
- 故障设备
- 维保计划

### 9.2 后续真实接入方式

真实 IoT 接入建议走独立链路：

```text
设备
→ Gateway
→ MQTT Broker
→ IoT Service
→ Event Bus
→ Device / Alert / Health / AI 服务
→ Admin / App 展示
```

### 9.3 Admin 侧承载方式

Admin 不应关心设备厂商协议，只关心标准化后的字段，例如：

- deviceId
- deviceType
- onlineStatus
- batteryLevel
- signalStrength
- lastSeenAt
- alertLevel
- dataPayloadSummary

## 11. 文档协同建议

建议后续将 docs 目录按如下方式协同使用：

- `PRODUCT_DESIGN.md`：定义产品结构、页面与三端视角
- `DESIGN_SYSTEM.md`：定义颜色、排版、卡片、导航、交互规范
- `PLATFORM_ARCHITECTURE.md`：定义平台级三端 + AI + IoT 目标架构
- `IMPLEMENTATION_BLUEPRINT.md`：定义当前工程如何承接平台方案

推荐使用方式：

1. 先用 `PLATFORM_ARCHITECTURE.md` 定义全局边界
2. 再用 `IMPLEMENTATION_BLUEPRINT.md` 判断当前仓库要做什么、不做什么
3. 然后用 `PRODUCT_DESIGN.md` 和 `DESIGN_SYSTEM.md` 指导页面与交互落地

这样可以避免出现“页面先做了，但领域边界和平台方向后面推翻重来”的问题。
