### 4.1.2.1 AI 驱动入住评估链路

在 Admin 管理后台内，入住办理不应只停留在表单录入，而应形成一条完整的 AI + 业务闭环：

```text
入住登记表填写完成
→ 系统汇总老人基础信息、病史、慢病、过敏史、ADL、认知、用药信息
→ AI 根据护理级别判别标准生成护理级别建议
→ 护理主管确认或调整建议级别
→ 系统按确认后的护理级别生成护理计划
→ 系统拆解为护理任务与定时提醒
→ 到点通知护理人员执行护理项
→ 执行结果回写数据库并继续反哺 AI
```

这条链路的目标是把“入住评估”从一次性录入动作升级为持续可执行的护理起点。

关键约束：

- AI 负责给出护理级别建议，不直接最终定级
- 最终护理级别必须由人工确认
- 护理计划由规则模板与 AI 建议共同生成
- 护理任务和提醒必须进入统一任务中心与消息中心
- 执行反馈要沉淀到数据库，作为后续风险识别和计划调整依据

## 2. 平台定位

### 2.1 核心定位

统一养老数字化平台（Elderly Care Digital Platform）面向养老机构、护理人员、家属和 AI 应用，提供统一的数据、统一的流程、统一的能力中台。

### 2.2 设计原则

| 维度 | 说明 |
|------|------|
| 三端统一 | Admin、员工、家属使用统一业务中台，不做端到端直连 |
| 数据驱动 | 护理、健康、报警、设备、费用全部沉淀为结构化数据 |
| AI 增强 | AI 不作为独立孤岛，而是嵌入护理、运营、健康、家属沟通四类场景 |
| IoT 融合 | 所有设备统一接入 IoT 平台，不直接耦合业务页面 |
| 可持续扩展 | 支持从单机构扩展到连锁机构、SaaS 多租户、区域化平台 |

### 2.3 平台边界原则

平台遵循一个关键原则：

**所有端都只访问中台，不直接互相通信。**

这意味着：

- 家属 APP 不直接访问员工任务系统
- 员工 APP 不直接访问家属端数据模型
- Admin 后台不直接连接设备协议层
- AI 应用不直接修改业务数据库，而是通过中台服务调用

## 3. 四层总体架构

推荐采用四层平台架构：

```text
应用层
│
├── Admin 管理后台
├── 家属 APP
├── 外勤员工 APP
└── AI 应用
│
业务中台
│
├── 老人管理
├── 护理服务
├── 健康医疗
├── 员工管理
├── 设备管理
├── 消息中心
├── 费用系统
│
数据中台
│
├── 运营数据
├── 健康数据
├── IoT 设备数据
├── AI 训练数据
│
基础设施层
│
├── API Gateway
├── IoT 平台
├── AI 平台
├── 消息系统
├── 数据仓库
```

### 3.1 应用层

应用层负责用户交互与场景承载，核心职责是：

- 展示信息
- 执行业务动作
- 消费中台接口
- 不承载核心业务规则

### 3.2 业务中台

业务中台负责统一承载养老业务能力，核心职责是：

- 统一领域模型
- 统一业务流程
- 统一接口输出
- 统一数据治理

### 3.3 数据中台

数据中台负责聚合业务数据、健康数据、设备数据和 AI 特征数据，核心职责是：

- 统一存储口径
- 统一指标口径
- 为报表、风控、AI 提供数据支撑

### 3.4 基础设施层

基础设施层负责平台可用性、弹性与集成能力，核心职责是：

- API 接入
- 消息分发
- 设备接入
- AI 服务调度
- 数据仓与分析引擎承载

## 4. 四端系统职责

## 4.1 Admin 管理后台

Admin 是机构运营中枢，服务对象主要包括：

- 院长
- 运营主管
- 护理主管
- 医护人员

核心功能建议如下。

### 4.1.1 养老院运营管理

模块建议：

- 老人管理
- 房间床位管理
- 入住管理
- 费用管理

价值：

- 统一运营视图
- 降低线下台账与 Excel 依赖
- 支撑多院区运营协同

### 4.1.2 护理服务管理

系统自动生成任务，例如：

```text
老人A
08:00 量血压
09:00 早餐护理
14:00 康复训练
20:00 服药
```

价值：

- 护理计划标准化
- 护理执行留痕
- 服务质量可量化

### 4.1.3 健康医疗系统

统一记录：

- 生命体征
- 医疗记录
- 用药记录

并支撑：

- 健康趋势追踪
- 风险识别
- 异常升级处理

### 4.1.4 IoT 设备管理

统一管理设备：

- 呼叫按钮
- 智能手环
- 跌倒检测
- 床位传感器

支持：

- 在线状态监控
- 设备资产台账
- 故障与维修闭环

### 4.1.5 报警中心

报警来源包括：

- 跌倒
- 健康异常
- 呼叫按钮
- 设备故障

平台要支持：

- 分级告警
- 升级链路
- 责任人响应
- 处理时效分析

### 4.1.6 数据分析

Dashboard 建议重点指标：

```text
入住率
护理完成率
员工效率
设备在线率
健康风险
```

## 5. 业务中台设计

建议业务中台按领域拆分为以下模块：

| 模块 | 核心职责 |
|------|----------|
| 老人管理 | 老人档案、入住、床位、标签、家属关系 |
| 护理服务 | 护理计划、任务生成、服务执行、打卡、交接 |
| 健康医疗 | 生命体征、病史、用药、医嘱、风险识别 |
| 员工管理 | 员工档案、角色权限、排班、绩效、任务归属 |
| 设备管理 | 设备接入、资产台账、在线状态、故障、维保 |
| 消息中心 | 站内信、推送、短信、家属通知、系统通知 |
| 费用系统 | 入住费用、护理套餐、账单、支付、欠费提醒 |

### 5.1 入住到护理执行的事件链

建议将“入住登记 → AI 分级 → 护理计划 → 提醒任务”设计为一条事件化链路：

```text
AdmissionSubmitted
→ AdmissionAssessed
→ CareLevelRecommendedByAI
→ CareLevelConfirmed
→ CarePlanGenerated
→ CareTasksScheduled
→ CareReminderTriggered
→ CareTaskExecuted
```

这样可以统一接入：

- 消息提醒
- AI 解释
- 任务中心
- 报表分析
- 审计追踪

### 5.1 业务中台设计原则

- 按领域拆分服务，而不是按前端页面拆服务
- 所有端共享统一领域对象定义
- 关键链路必须具备事件化能力，方便接入消息、报表与 AI
- 高风险动作必须保留责任人和审计信息

## 7. AI 平台设计

建议建立独立的养老 AI 平台层，作为业务中台和数据中台之上的增强能力。

### 7.1 AI 健康预测

目标：

- 健康风险预测
- 跌倒风险预测
- 突发疾病风险预测

例如：

```text
AI 预测：老人A
未来7天跌倒风险：高
```

### 7.2 AI 护理助手

为员工侧提供：

- 护理计划推荐
- 护理注意事项提醒
- 高风险任务优先级建议

### 7.3 AI 家属问答助手

家属可提问：

```text
妈妈今天状态怎么样？
```

AI 可汇总回答：

```text
今天心率正常
睡眠7小时
完成3项护理
```

### 7.4 AI 异常检测

自动识别：

- 健康异常
- 设备异常
- 行为异常

### 7.5 AI 平台建议架构

```text
AI 数据平台
   ↓
特征工程
   ↓
模型训练
   ↓
AI 服务 API
   ↓
Admin / 家属 APP / 员工 APP / 消息系统
```

### 7.6 AI 模型方向

- 健康风险预测模型
- 跌倒风险预测模型
- 行为分析模型
- 家属问答检索增强模型
- 护理助手决策建议模型

## 8. IoT 平台设计

设备统一接入 IoT 平台，不允许设备协议直接耦合业务系统。

### 8.1 设备类型

- 智能手环
- 血压设备
- 呼叫按钮
- 跌倒检测设备
- 床位传感器

### 8.2 协议建议

建议优先采用：

- MQTT 作为主协议

必要时可扩展：

- HTTP
- WebSocket
- 厂商私有协议适配层

### 8.3 IoT 接入链路

```text
设备
 ↓
IoT Gateway
 ↓
MQTT Broker
 ↓
IoT 平台
 ↓
数据中台
 ↓
业务中台 / 报警中心 / AI 平台
```

### 8.4 IoT 平台核心能力

- 设备注册与认证
- 心跳与在线监控
- 数据采集与标准化
- 规则引擎与告警
- 固件升级与运维管理

## 12. 最终形态架构图

```text
家属 APP
员工 APP
Admin 后台
AI 助手
      │
API Gateway
      │
业务中台
      │
数据中台
      │
AI 平台
      │
IoT 平台
```

### 12.1 最终形态说明

这是一个典型的医疗 SaaS + IoT + AI 平台化架构：

- 应用层承接场景
- 业务中台统一规则
- 数据中台沉淀资产
- AI 平台负责智能增强
- IoT 平台负责实时感知

在这个架构下，养老机构从“信息化工具”升级为“数据驱动 + 智能增强”的养老服务平台。
